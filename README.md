# HEDIS Analytics — Claims-Based Measure Implementation

**Project page:** https://zk-praxis.xyz/hedis.html

A working implementation of HEDIS quality measures using CMS Synthetic
Medicare claims data, built on Microsoft SQL Server.

The project takes raw CMS Fee-for-Service and Part D Event files,
loads them into a relational database, and produces member-level results
for a set of claims-based HEDIS 2024 measures for measurement year 2021.

---

## Stack

- **Database:** Microsoft SQL Server 2022 Developer Edition (Linux)
- **SQL client:** sqlcmd (mssql-tools18)
- **ETL:** Python 3 + pyodbc + pandas
- **Notebook environment:** JupyterLab
- **Source data:** [CMS Synthetic Medicare Enrollment, FFS Claims, and PDE Dataset](https://data.cms.gov/collection/synthetic-medicare-enrollment-fee-for-service-claims-and-prescription-drug-event)
  — same structure as real Medicare claims, generated by Synthea.

---

## Why this project

The CMS synthetic dataset mirrors real Medicare claims exactly in structure
and column naming. Implementing HEDIS measures with this dataset
demonstrates the same data engineering and SQL work that a health plan
analytics team does day to day:

1. Ingest raw claims and enrollment files from CMS
2. Build a relational schema that matches the source column names
3. Load and validate the data
4. Identify which measures can be computed from claims alone vs which
   require EHR, lab, or survey data
5. Implement each measure with proper denominator/numerator logic
6. Produce member-level outputs that can feed downstream reporting

## Measures implemented

| Measure | Name | Denominator | Rate |
|---|---|---|---|
| AAB | Avoidance of Antibiotics for Acute Bronchitis | 825 episodes | 98.3% |
| AMR | Asthma Medication Ratio | 59 | 78.0% |
| COL | Colorectal Cancer Screening | 4,860 | 26.6% |
| FUH | Follow-Up After Hospitalization for Mental Illness | 19 discharges | 68.4% / 84.2% |
| FUM | Follow-Up After ED Visit for Mental Illness | 0 | N/A — ED codes absent |
| IET | Initiation and Engagement of SUD Treatment | 168 episodes | 100% init / 6.5% engagement |
| PCR | Plan All-Cause Readmissions | 3,049 admissions | 31.9% |

### Measures evaluated and dropped

| Measure | Reason |
|---|---|
| ABA — Adult BMI Assessment | Z68.x BMI codes absent from synthetic data |
| BCS — Breast Cancer Screening | Mammography HCPCS nearly absent |
| CBP — Controlling High Blood Pressure | BP readings are EHR data, not claims |
| CDC — HbA1c Testing | Lab CPT codes (83036, 83037) absent |
| LBP — Imaging for Low Back Pain | Lumbar imaging codes absent |
| PPC — Prenatal/Postpartum Care | Delivery CPT codes absent |
| SAA — Antipsychotic Adherence | Schizophrenia dx (F20-F29) absent |
| URI — Appropriate Treatment for URI | Population too thin (~40 members) |

These exclusions are documented because they are themselves a finding —
claims-based data can support a meaningful subset of HEDIS, but lab,
medication, and EHR data are required for the rest. The `testing/measures.ipynb`
notebook contains the coverage queries that drove each decision.

## Reproducing the build

The full sequence of commands used to set up the environment, load the data,
and run each measure is captured in `COMMAND_LOG.md`. The narrative version
with decisions and problems encountered is in `DEVLOG.md`.

High-level steps:

1. Install SQL Server 2022 Developer Edition and `mssql-tools18` (`sqlcmd`)
2. Download the CMS synthetic FFS, Beneficiary 2021, and PDE files into
   `data/raw/`
3. Run `schema/create_tables.sql` via `sqlcmd` to create the `hedis` database
4. Run `etl/load_cms_data.py` to bulk-load all CSV files
5. Use `testing/db_verification.ipynb` to confirm row counts and date ranges
6. Run any measure `.sql` file via `sqlcmd` to produce member-level output,
   or open `measures/measures.ipynb` to run all measures and export results to `results/*.csv`

## Design notes

- **Column naming:** every table column matches the source CMS column name
  exactly. No snake_case translation, no renaming. This keeps the schema
  recognizable to anyone who has worked with CMS data.
- **Separation of concerns:** `schema/` is T-SQL only, `etl/` is Python only,
  `measures/` contains one `.sql` per measure (standalone, runnable via `sqlcmd`)
  and a single combined `measures.ipynb` with all measures, a TOC, and export cells.
- **Each measure has a header comment block** with denominator, numerator,
  data sources, measurement year, and direction, in plain English.
- **Member-level output:** every measure produces one row per index event
  with binary numerator flags, not just an aggregate rate. This is how a
  health plan would actually use the data downstream.
- **Honest about data limits:** several measures (AAB, AMR) require NCQA
  NDC value sets that are not bundled with the synthetic data. Those queries
  use documented placeholders and explicitly call out the limitation in both
  the SQL header and notebook conclusion.
